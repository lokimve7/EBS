# Task Writing

## 목적
이 문서는 task 문서를 작성하거나 구현 후 갱신할 때 따라야 하는 규칙입니다.

## 기본 원칙

- task 문서는 현재 단계에서 필요한 결과와 검증 기준을 고정하는 문서입니다.
- task 문서에는 기본적으로 What, 즉 무엇이 필요한지를 적습니다.
- 구현 전 task 문서에는 How, 즉 내부 구현 방식, 파일 분리 방식, 함수 구조를 추측해서 적지 않습니다.
    - 단, 사용자의 요청에 How가 포함된 경우 해당 내용을 task 문서에 그대로 반영한다.
- 구현 후에는 실제 추가된 파일명, 버튼명, 섹션명, 저장 경로 등과 같은 구현 결과를 기준으로 아래 `수동 작업 작성 규칙`과 `결과 확인 작성 규칙`에 맞춰 task 문서의 `수동 작업`과 `결과 확인` 항목을 갱신합니다.
- 사용자 요청과 현재 task 범위를 벗어나는 항목은 task 문서에 추가하지 않습니다.
- 새 task 문서를 생성하면 `docs/PLANS.md` 작업리스트의 해당 단계 `Task 문서` 칸도 생성한 task 문서 링크로 갱신합니다.

## 작성 시점별 규칙

### 구현 전

- `구현 항목`에는 현재 task에서 달성해야 하는 결과만 적습니다.
- `범위 밖`에는 이번 task에서 일부러 다루지 않을 항목을 적습니다.
- `사전 전제`에는 이전 단계 산출물이 필요할 때만 실제 전제 대상을 적습니다.
- `수동 작업` 과 `결과 확인`은 실제 코드 구현 전에는 구체 내용을 작성하지 않습니다.
- 실제 이름을 알 수 없는 파일명, 버튼 텍스트, 상태 문구를 추측해서 구체화하지 않습니다.
- 요청에 버튼명이나 UI 요소 이름이 명시되어 있으면 해당 이름으로 코드베이스를 검색해 이미 존재하는지 확인한 뒤 구현 항목을 작성합니다.

### 구현 후

- 완료된 `구현 항목`과 `에이전트 확인` 체크박스를 실제 완료 범위만 `[x]`로 갱신합니다.
- `수동 작업`은 실제로 사람이 결과를 만들거나 진행시키기 위해 해야 하는 조작을 기준으로 구체화합니다.
- `결과 확인`은 실제 결과가 기대한 상태인지 사람이 확인할 수 있는 항목을 기준으로 구체화합니다.
- 사용자가 수동 작업 또는 결과 확인 완료를 응답하기 전에는 해당 확인 항목을 완료 처리하지 않습니다.
- 진행 중인 task에서 추가 변경이 발생해도 기존에 실제 완료된 항목은 유지하고, 새 변경으로 인해 새로 확인해야 하는 항목만 `[ ]`로 둡니다.

## 수동 작업 작성 규칙

`수동 작업`에는 에이전트가 대신할 수 없어서 사람이 실행 전에 미리 준비해야 하거나 직접 입력, 연결, 선택, 설정해야 하는 항목만 적습니다. 초보자가 그대로 따라 할 수 있도록 단계별로 구체적으로 적습니다.

- 결과를 보기 위한 실행 절차나 확인 절차는 `수동 작업`에 적지 않고 `결과 확인`에 적습니다.

아래는 대표적인 예시입니다.

- `.env`에 추가할 키 이름
- 외부 서비스 계정 생성 또는 로그인
- 파일에 직접 붙여넣을 값의 종류
- 사람이 직접 선택하거나 연결해야 하는 파일, 값, 경로
- 레이아웃 이미지 또는 참고 이미지 첨부

포함하지 않을 내용:

- 서버 실행을 위해 파일을 더블클릭하는 동작
- 브라우저에서 URL을 여는 동작
- 페이지 접속 또는 버튼 클릭
- 에이전트가 대신 수행할 코드 수정
- 현재 task와 무관한 다음 단계 준비
- 대상과 방법이 없는 추상 문장

### 좋은 예

- `.env` 파일에 `OPENAI_API_KEY` 값을 입력한다.
- 관리자 페이지에 로그인한 뒤 발급받은 사이트 키를 설정 화면에 붙여넣는다.
- `assets/images/`에 사용할 참고 이미지를 직접 넣는다.

## 결과 확인 작성 규칙

`결과 확인`에는 사람이 task 결과를 확인하기 위해 실제로 수행하는 실행 절차와, 그 결과가 기대한 상태인지 확인하는 항목을 적습니다. 초보자가 그대로 따라 할 수 있도록 단계별로 구체적으로 적습니다.

- 결과 확인을 위해 필요한 실행 동작은 `결과 확인`에 포함할 수 있습니다.
- `수동 작업`에 적은 입력, 연결, 설정 절차를 같은 의미로 반복하지 않습니다.

사람이 실행하고 눈으로 확인할 수 있는 모든 절차를 적습니다. 아래는 대표적인 예시입니다.

- 파일 더블클릭
- 브라우저에서 URL 열기
- 페이지 접속
- 버튼 클릭
- 로딩 상태 확인
- 성공/실패 메시지 확인
- 카드, 리스트, 미리보기, 저장 결과 확인
- 실행 후 기대한 화면이나 동작이 나오는지 확인

포함하지 않을 내용:

- 코드 내부 구조 설명
- 구현되지 않은 다음 단계 동작
- 현재 task 범위 밖의 통합 흐름

### 좋은 예

- `start-server.bat`을 더블클릭했을 때 서버 실행 창이 열리는지 확인한다.
- 브라우저에서 `http://localhost:3001`을 열었을 때 페이지에 정상 접속되는지 확인한다.
- `http://localhost:3001/01_script.html`에 접속했을 때 레이아웃 이미지와 유사한 화면이 보이는지 확인한다.
- `대본 생성` 버튼을 눌렀을 때 주제가 비어 있으면 안내 문구가 보이는지 확인한다.
- 씬 카드 하나의 `이 씬만 다시 만들기` 버튼을 눌렀을 때 해당 카드에만 로딩 표시가 보이는지 확인한다.
