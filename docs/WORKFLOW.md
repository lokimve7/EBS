# Workflow

## 목적

이 문서는 AI 에이전트가 task를 진행할 때 따라야 할 작업 순서, 진행 제약, 상태 관리 규칙을 정의합니다.

## 문서 우선순위

문서 해석 우선순위는 아래와 같습니다.

1. 사용자 직접 요청
2. 현재 진행 중인 task 문서
3. `docs/PLANS.md`
4. 나머지 프로젝트 문서

문서끼리 충돌하거나 현재 저장소 상태와 맞지 않으면, 구현 전에 불일치를 먼저 보고하고 수정 범위를 제안합니다.

## 작업 흐름

사용자가 task 초안을 기반으로 진행을 요청하면 아래 순서로 작업합니다.
아래 순서는 형식적으로 진행하지 않습니다. 정확하게 진행합니다.

1. 관련 task와 선행 산출물을 확인합니다.
2. `docs/PLANS.md`에서 현재 단계 상태를 확인하고 작업 시작 시 `진행중`으로 갱신합니다.
3. `docs/ARCHITECTURE.md`의 내용을 모른다면 확인합니다.
4. 관련 코드와 현재 화면 구조를 읽습니다. 관련 코드가 아직 없으면 이 단계는 건너뜁니다.
5. `docs/STYLEGUIDE.md`의 내용을 모른다면 확인합니다.
6. 아래 항목들을 보수적으로 판단하며, 하나라도 해당되거나 조금이라도 의심되면 `책임 범위 의심 파일`에 기재합니다.
    - 같은 성격의 내용이 여러 파일에 반복되는지
    - 공통화 가능성이 조금이라도 있는지
    - 추가할 코드의 성격이 대상 파일의 현재 책임 범위와 다른지
    - 의심 파일 기재는 분리 필요 판단과 별개이며, 조금이라도 의심되면 먼저 기록한다.
    
7. 구현 전 반드시 아래 블록을 출력하고 사용자 승인을 받은 뒤 다음 단계로 넘어갑니다.

**[구현 진행 승인 요청]**
- 생성 후보 파일:
- 변경 후보 파일:
- 변경 요약:
- 책임 범위 의심 파일: (조금이라도 의심되면 기재, 분리 여부는 사용자가 결정, 의심이 전혀 없으면 "없음")
- 의심 사유: (없으면 "없음")
- 의심에 대한 제안 : (의심 파일마다 분리안과 기존 파일 유지안을 각각 설명하고 사용자가 선택하게 한다. 이 항목 작성이 의심 해소를 뜻하지 않는다.)
- 의심에 대한 추천 진행 : (분리안/유지안, 간단한 이유)
- 사용자 승인 전 구현 안 함

8. 필요한 코드 변경을 구현합니다.
9. 구현 후 `docs/TASK_WRITING.md`의 작성 규칙에 맞춰 task 문서를 실제 코드 기준으로 갱신합니다.
10. 구현 전 승인 요청의 `책임 범위 의심 파일`은 구현 후 상태에 따라 `Pending Issues 관리` 규칙에 맞게 정리합니다.
11. `docs/TESTING.md`의 현재 실행 확인 모드에 따라 확인을 진행합니다.
12. 사용자에게 필요한 수동 작업 또는 결과 확인을 요청합니다.
13. 결과 확인이 끝나면 아래 `작업 로그 관리` 규칙에 따라 로그를 정리합니다.
14. 확인이 완료된 뒤 task 문서와 `docs/PLANS.md`에서 현재 단계 상태를 `완료`로 갱신합니다.

## 진행 제약

- 구현은 task 구현 항목에 명시된 것만 합니다. 범위 밖 항목은 구현하지 않습니다.
- 프로젝트 내 민감한 계정 정보, API 키, 인증 토큰, 비밀값은 읽거나 출력하지 않습니다.
- 결과 확인이 끝나기 전에는 다음 단계로 넘어가지 않습니다.
- 구현과 문서 갱신은 같은 작업 흐름 안에서 함께 반영합니다.

## 상태 관리 원칙
- 최근 작업 로그에는 구현, 실행 확인, 결과 확인 결과를 남깁니다.
- `docs/PLANS.md`, `docs/LOGS.md`, `docs/PENDING_ISSUES.md` 같은 기록 문서의 운영 규칙은 이 문서를 기준으로 관리합니다.

## Pending Issues 관리
- 해소되지 않은 의심은 `docs/PENDING_ISSUES.md`에 기록합니다.
- 이번 작업에서 기존 구조를 유지하기로 했더라도, 그 결정이 의심 해소를 뜻하지 않으면 `해소되지 않은 의심`으로 보고 `docs/PENDING_ISSUES.md`에 기록합니다.
- 같은 파일과 같은 성격의 의심은 중복으로 추가하지 않습니다.
- 실제로 해소된 항목은 `docs/PENDING_ISSUES.md`에서 제거합니다.

## 작업 로그 관리

- `docs/PLANS.md`의 `최근 작업 로그`는 최신 진행 상황 확인용으로 유지합니다.
- `docs/PLANS.md`의 `최근 작업 로그`가 20줄 이상이 되면, 작업 중인 에이전트는 가장 오래된 15줄을 `docs/LOGS.md`로 이동하고 최신 5줄만 남깁니다.
- 이동한 로그는 삭제하지 않고 `docs/LOGS.md`에 유지합니다.
